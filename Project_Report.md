## ABSTRACT
	 
A Hospital Management System (HMS) is a comprehensive software solution designed to streamline and automate the daily operations of a healthcare facility. This project focuses on developing an efficient and user-friendly system that manages patient records, doctor information, appointments, billing, and administrative tasks in a centralized digital environment.

The system eliminates the need for manual record-keeping, thereby reducing errors, improving data accuracy, and saving time for hospital staff. It provides functionalities such as patient registration, appointment scheduling, medical record management, and billing generation. The system ensures secure storage of sensitive data and allows authorized users to access and update information in real-time.

This project is developed using modern web technologies featuring a React JS frontend, Node.js REST API backend, MongoDB NoSQL Database, and a Python-powered Machine Learning AI Engine to ensure scalability, reliability, and modern performance. The implementation of this system enhances operational efficiency, improves patient care, and supports better decision-making within the hospital. 

In conclusion, the Hospital Management System offers a cost-effective and scalable solution for managing complex hospital activities, significantly reducing administrative workload, and improving overall healthcare service quality.

## 5. Coding Operations & Implementation

### a. Database Creation & Schemas
Because MongoDB handles collections dynamically, traditional rigid SQL `CREATE TABLE` commands are bypassed in favor of Mongoose ODM Schemas built into the Express layer:

```javascript
// Sample: Invoice.js Schema containing object nesting
const invoiceItemSchema = new mongoose.Schema({
    description: String,
    quantity: { type: Number, default: 1 },
    unit_price: { type: Number, default: 0 },
    total: { type: Number, default: 0 },
}, { _id: false });

const invoiceSchema = new mongoose.Schema({
    invoice_number: { type: String, unique: true },
    patient: { type: mongoose.Schema.Types.ObjectId, ref: "PatientProfile", required: true },
    items: { type: [invoiceItemSchema], default: [] },
    subtotal: { type: Number, default: 0 },
    status: { type: String, enum: ["unpaid", "paid", "partial", "cancelled"], default: "unpaid" }
}, { timestamps: true });

// Auto-generating unique alphanumeric invoice serial increments
invoiceSchema.pre("save", async function () {
    if (!this.invoice_number) {
        const count = await mongoose.model("Invoice").countDocuments();
        this.invoice_number = `INV-${String(count + 1).padStart(4, "0")}`;
    }
});
```

### b. Complete Project Coding Flow
1. **Server Initialization**: `server.js` triggers Node.js to spin up Express app endpoints on port 8000 and establishes connecting to the MongoDB database string via `mongoose.connect()`.
2. **Routing Traffic**: `app.use('/api/patients', require('./routes/patients'));` maps specific base-urls to specific modular files reducing monolithic code bloat.
3. **Execution Controllers**: The routes trigger controller logic which queries the database, applies business logic (e.g. calculating billable hours), formats JSON, and replies to the Frontend HTTP request.

- **Code Efficiency**: To reduce unnecessary waterfall delays when loading dashboards, Mongoose queries use `.populate()` logic combining what would be several single queries into one bulk SQL-like joined payload.

```javascript
// Efficient Authorization Middleware Example
const verifyToken = (roles = []) => {
    return (req, res, next) => {
        const authHeader = req.headers.authorization;
        if (!authHeader) return res.status(401).json({ msg: "No token, authorization denied" });
        const token = authHeader.split(" ")[1];
        try {
            const decoded = jwt.verify(token, process.env.JWT_SECRET);
            if (roles.length && !roles.includes(decoded.role)) {
                return res.status(403).json({ msg: "Permission denied for this role type." });
            }
            req.user = decoded;
            next();
        } catch (err) {
            res.status(401).json({ msg: "Token format invalid or expired.", error: err.message });
        }
    };
};
```

## 8. System Security Measures

### a. Data Encryption
Under no circumstances are user passwords saved bare in the datastore. The system leverages `bcryptjs` doing high-entropy iterative cryptographic hashing upon user creation.

### b. Access Rights
Through JSON Web Tokens (JWT), each logged-in user carries a cryptographic signature verifying their identity and `user.role`. This signature must be submitted to the backend as a Bearer Header on every API action attempting to mutate data.
